  Test-NetConnection -ComputerName 136.119.28.83 -Port 5432

backend building plan phases
You are a senior backend engineer. Build a backend-only MVP for a hackathon project called:

Smart Resource Allocation — Data-Driven Volunteer Coordination for Social Impact

Goal:
Build a working backend prototype where NGOs can:
1. upload a blank paper form image,
2. extract candidate fields using AI,
3. map those fields to a master field catalog,
4. generate a draft survey form,
5. review and publish the form,
6. submit surveys using that form,
7. analyze submitted survey data into urgent community needs,
8. match volunteers to those needs based on skills, availability, and location.

Important:
This is a prototype. Keep the design simple, correct, and demoable. Do not overengineer. Use practical defaults and make reasonable implementation decisions without asking too many questions.

==================================================

Use:
- Node.js
- Express.js

- Firebase Auth / Google Identity Platform for authentication
- Vertex AI / Gemini service abstraction for AI features

Preferred implementation choices:
- TypeScript preferred; plain JavaScript is acceptable if it speeds up delivery
- Use Knex or Sequelize for migrations and DB access
- Use Firebase Admin SDK for token verification
- Use Google Cloud client libraries for GCS
- Use dotenv for config
- Use Joi/Zod/express-validator for request validation
- Use multer only if needed for local multipart fallback, but prefer signed GCS upload URL flow

Architecture principles:
- simple MVP first
- multi-tenant by org_id
- one user belongs to one NGO for prototype
- dynamic forms with template versioning
- EAV response storage for flexible survey fields
- AI pipeline must be async-capable
- all cloud and AI integrations must be behind service classes so mock mode is possible

==================================================
2. SIMPLE PROTOTYPE DESIGN DECISIONS
==================================================

Use these simplified rules:
- One user belongs to one organization
- Role is stored directly in users.role
- Supported roles: superadmin, ngo_admin, field_worker, volunteer
- One user may optionally have one volunteer profile
- Location can be stored as:
  - location_text
  - latitude
  - longitude
- Keep document review simple with a status field
- Matching logic can be weighted scoring:
  - skill overlap
  - availability
  - location proximity
- Keep AI services abstracted so real integrations can be swapped with stub/mock mode
- Keep one main database, no microservices
- No websockets needed
- No advanced geospatial engine required for MVP

8. AI FLOW TO SUPPORT
==================================================

Support this full workflow:

1. NGO admin requests signed upload URL for a blank form image
2. frontend uploads file directly to GCS
3. backend stores metadata in documents
4. admin triggers extraction
5. AI extracts candidate fields from uploaded form
6. AI maps extracted fields to field_catalog when possible
7. backend creates a draft form_template + form_template_version + form_fields
8. admin reviews and edits fields
9. admin publishes the version
10. field worker submits surveys using that version
11. backend stores survey answers in survey_responses EAV format
12. AI analyzes survey and creates needs_analysis records
13. AI or business logic maps required skills
14. matching service ranks volunteers
15. NGO admin assigns a volunteer
16. assignment status can be updated

10. BUSINESS LOGIC REQUIREMENTS
==================================================

Implement these important rules:

- form_template_version must be immutable after publish
- survey must always reference a specific template_version_id
- survey responses must reference form_field_id from that version
- only one published version per form template at a time
- when publishing a new version, previous published version can remain historical but not editable
- EAV response insert/update must validate value type against input_type
- volunteer matching must return explanation details, not just a score
- tenant guard must prevent cross-org access
- document extraction should not directly auto-publish forms
- extracted forms should start as draft/review state
